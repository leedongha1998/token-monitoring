# Definition of Done (작업 완료 기준)

이 문서는 AI 에이전트와 개발자 모두에게 적용되는 **작업 완료의 유일한 기준**이다.
아래 기준을 전부 충족하기 전에는 어떤 작업도 "완료"로 선언하지 않는다.
"컴파일 성공"은 완료가 아니다. "구현했다"는 보고는 완료가 아니다.

## 1. 완료 조건 (전부 충족해야 함)

다음 명령이 **모두 성공**해야 한다:

```bash
./gradlew spotlessCheck   # 코드 포맷 검사
./gradlew build           # 컴파일 + 전체 테스트 + ArchUnit 테스트
```

추가로:

- [ ] 신규/수정된 public 메서드에 대응하는 테스트가 존재한다
- [ ] DB 스키마 변경이 있다면 Flyway 마이그레이션 파일(`V{n}__설명.sql`)이 함께 작성되었다
- [ ] API 변경이 있다면 `docs/rules/05-api-design.md`의 응답 포맷을 따른다
- [ ] 코드에 `TODO`, `FIXME`, 주석 처리된 코드 블록을 남기지 않았다
- [ ] 커밋 메시지가 `docs/rules/03-git-convention.md`를 따른다

## 2. 절대 금지 사항

작업을 "완료"시키기 위해 다음 행위를 하는 것을 금지한다.
이 항목을 위반하느니 **작업을 미완료 상태로 보고하는 것이 옳다.**

- 실패하는 테스트를 통과시키기 위해 **테스트 코드를 수정·약화·삭제**하는 행위
- `@Disabled`, `@Ignore`로 테스트를 비활성화하는 행위
- ArchUnit 규칙 테스트를 수정하거나 예외를 추가하는 행위
- `spring.jpa.hibernate.ddl-auto` 값을 변경하는 행위 (validate 고정)
- `build.gradle`에 **새 의존성을 추가**하는 행위 (사용자 승인 필수)
- Spotless/Checkstyle 설정을 완화하는 행위
- CI 설정 파일, `docker-compose.yml`, `.env*` 파일을 수정하는 행위

## 3. 미완료 보고 형식

기준을 충족하지 못한 채 작업을 종료해야 한다면, 다음 형식으로 보고한다:

```
[미완료] 사유: {실패한 검증 단계}
- 시도한 것: ...
- 실패 원인 분석: ...
- 다음 단계 제안: ...
```

거짓 완료 보고("모든 테스트가 통과했습니다")는 가장 심각한 위반이다.
완료 선언 전에 반드시 위 검증 명령을 **실제로 실행하고 출력을 확인**한다.

## 4. 작업 범위 준수

- 요청받은 범위 밖의 파일을 수정하지 않는다 (리팩토링 욕구 억제)
- 범위 밖 개선이 필요해 보이면 코드를 고치지 말고 **제안만** 보고한다
- 하나의 작업 단위 = 하나의 커밋 단위를 유지한다
